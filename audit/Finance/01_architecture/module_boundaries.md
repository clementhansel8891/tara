## Module Boundaries: Finance Core

### 1. Internal Boundaries
The Finance module is subdivided into three distinct domains that must maintain separation to ensure scalability and auditability.

#### Core Ledger Domain
- **Scope**: Double-entry consistency, Journal entries, Ledger atomicity.
- **Gatekeepers**: `LedgerPostingService`, `PostingGatewayService`.
- **Invariants**: 
  - SUM(DEBIT) == SUM(CREDIT) for every Journal.
  - Ledger entries are immutable once POSHTED.

#### Fiscal Domain
- **Scope**: Accounting periods, COA management, Fiscal years.
- **Gatekeepers**: `FiscalPeriodService`, `ChartOfAccountService`.
- **Invariants**:
  - No postings allowed to CLOSED periods.
  - Period closing requires zero trial balance variance.

#### Reporting Domain
- **Scope**: P&L, Balance Sheet, Forecasts.
- **Gatekeepers**: `ReportingEngineService`, `FinancialReportService`.
- **Dependencies**: Depends on snapshots generated by the Fiscal Domain.

### 2. External Boundaries (Integrations)

#### Upstream (Subledgers -> Finance)
- **AR (Accounts Receivable)**: Communicates via the `PostingGateway`.
- **Inventory**: Communicates via `InventoryAccountingIntegrationService`.
- **Payroll**: Direct integration with the `PayrollRepository`.

#### Downstream (Finance -> Infrastructure)
- **Prisma/DB**: All persistence flows through the Repository Layer.
- **Audit Engine**: All state changes trigger events to the `AuditModule`.

### 3. Boundary Violations Detected
> [!NOTE]
> Currently, the `FinanceModule` imports `ArModule` directly. This is acceptable for a "Core" module, but long-term scaling may require an Event-Bus approach to decouple further.
